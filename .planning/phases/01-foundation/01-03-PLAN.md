<!-- Multi-stage Dockerfile from RESEARCH.md Pattern 5 — use this exactly. -->

```dockerfile
# Stage 1 — Build React frontend
FROM node:20-slim AS frontend
WORKDIR /app/frontend
COPY frontend/package*.json ./
RUN npm ci
COPY frontend/ .
RUN npm run build

# Stage 2 — Run FastAPI backend
FROM python:3.12-slim
WORKDIR /app

COPY backend/requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY backend/ .
COPY --from=frontend /app/frontend/dist ./static

EXPOSE 8080
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]
```

Key decisions embedded in the Dockerfile:
- node:20-slim (not node:20 — smaller image, D-08)
- python:3.12-slim (not 3.14 which is installed locally — D-08, RESEARCH Standard Stack)
- npm ci (not npm install — reproducible installs from package-lock.json)
- pip install --no-cache-dir (smaller image layer)
- uvicorn hardcoded to port 8080 (Railway PORT env var must also be set to 8080 in dashboard)
- CMD exec form ["uvicorn", ...] not shell form (cleaner process management)

Railway PORT note: Railway injects $PORT. Our Dockerfile hardcodes 8080. Set PORT=8080 in Railway env vars to align. Alternatively use shell CMD form with ${PORT:-8080} — but hardcoding 8080 is simpler and matches task.md canonical spec.

Stage 1 (node:20-slim AS frontend):
- WORKDIR /app/frontend
- COPY frontend/package*.json ./ then RUN npm ci (not npm install)
- COPY frontend/ . then RUN npm run build
- Output: /app/frontend/dist/ populated with the compiled React app

Stage 2 (python:3.12-slim):
- WORKDIR /app
- COPY backend/requirements.txt . then RUN pip install --no-cache-dir -r requirements.txt
- COPY backend/ . (copies main.py, config.py, db.py into /app)

- EXPOSE 8080
- CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8080"]

CRITICAL: The COPY backend/ . step puts main.py in /app/main.py. The CMD "main:app" must match this — uvicorn looks for main.py in the working directory (/app). Verify WORKDIR is /app before the CMD.

Do NOT add any ENV statements with secret values. All configuration comes from environment variables at runtime (DEPLOY-04).

After creating the Dockerfile, verify the build locally:
```bash
docker build -t jarvis-test . 2>&1 | tail -20
```

If docker is not available locally, skip the docker build step — Railway will be the first real build test.
